# 事务、隔离级别及对性能的影响

## 1. 数据库事务

> 构成单一逻辑操作单元的一组操作集合称作一个事务。

从外部看，数据库的操作一般是单一操作，比如A资金账户减少100元，B资金账户增加100元。但是从代码实现层面和数据库内部看，这涉及到一组操作。 一组操作构成一个事务来执行时，数据库需要保证，要么几个操作全部成功，要么一个操作也不要做。

数据库事务的性质：

1. 原子性：事务中的操作，要么在数据库中全部正确地反映出来，要么全不反映出来。
2. 一致性：隔离执行事务时（换言之，在没有其他事务并发执行时）保持数据库的一致性。
3. 隔离性：尽管多个事务可以并发执行，但数据库保证，对于任何事务Ti和Tj，在Ti看来，Tj要么在Ti开始之前已经全部完成，要么在Ti之后才开始执行。
4. 持久性：事务成功执行后，对数据库中记录的改变是永久的，即使在事务执行过程中发生了系统故障。

上述四种性质简称为`ACID`特性。后面我们会看到，确保`ACID` 特性可能对数据库的性能产生较大的影响。因此，在代码层面，必须了解不同隔离级别下的影响面，在性能和数据安全两方面做出妥协和折中。

## 2. 事务执行中会发生什么问题？

在代码执行过程中使用错误的隔离模式，有可能造成比较严重且难以定位的问题，而这类问题主要还得从代码逻辑上来分析。 应该说，开发者应该在代码设计时，提前分析并发场景下数据库行为，而不是等数据库中的数据出了问题，在回来排查。

错误的隔离级别可能产生什么样的问题？知乎上有相当多的讨论，可以自行查询参考。

1. 脏读(Dirty Read)： 想象有个事务T1正在将某个值修改但为提交到数据库，此时T2事务读取了T1修改后的值，并做了更新。之后T1事务发生回滚，撤销修改。这是T2读到的值就是无效的。脏读一般发生在`UPDATE`操作。
2. 幻读（**Phantom Read** ）：事务T1根据某个筛选条件正在操作数据库中某一部分数据，此时事务T2正在向数据库中插入数据，并且事务T2插入的数据能够匹配事务T1的筛选条件。当两个事务并发执行时，有可能事务T1获取的操作数据集没有包含T2新插入的记录，当T1提交后，T2提交的新数据没有被T1执行到。 
3. 不可重复读（**Non Repeatable read**）：当事务T1对某行记录做多次读取时，事务T2正在对此行记录进行修改，对T1来说，它多次读取的数据不一致，即不可重复读。

## 3. 事务的隔离级别

按照ANSI标准，数据库事务的隔离级别有四种：`READ UNCOMMITTED`(读未提交)、`READ COMMITTED`(读提交)、`REPEATABLE READ`

(可重复读)、`SERIALIZABLE`(序列化)

事务的隔离级别是怎么划分出来的？简单概括，是根据一个事务的操作在其他事务中的可见性来区分。 事务需要使用`begin`语句开启，使用`commit`语句提交，在他们两者之间的部分是这个事务的操作。 当多个事务并发执行时，一个事务能不能在他自己的操作过程中看到其他事务已做但尚未提交的操作，决定了各个事务之间的隔离程度。

1. `READ UNCOMMITTED`:一个正在执行的事务，可以看到其他事务尚未提交的操作。（隔离程度最低）
2. `READ COMMITTED`: 一个事务只能在其他事务执行完`commit`语句后，才能看到其他事务的操作。（隔离程度稍微高一点，并发的事务不能互相看到对方的操作）
3. `REPEATABLE READ`:  可重复读保证多个并发事务在数据库中看到的数据行是相同的（屏蔽了插入删除操作？）。对于postgresql来说，可重复读事实上是创建一个当前表的快照，在事务内的所有读操作都发生在这同一个快照上，避免其他事务在某个点提交对本事务的影响。需要特别说明的是：这里的可重复读并非通常理解意义上的”可以重复读取数据“的概念，通过”快照“这个词的含义大家可以理解到：针对当前本事务来说，所使用的快照就这一份，所以本事务可以在这份快照上重复读。一个可重复读的事务可能在两次执行获得的数据不一样（因为使用了两份不同的快照）。总结来说，可重复读没有避免”幻读“。
4. `SERIALIZABLE`:序列化，数据库最高级别的隔离，所有事务串行执行。

## 4. 隔离级别在解决什么事情？ 它的意义是什么？

对于开发者来说，公认的一点是：**在高并发场景下，保证数据一致性很困难**，除非你认为效率和延迟不是严重问题，粗粒度锁也能应付业务需求，否则数据一致性的保证就需要你，即了解数据库的隔离行为，也要深刻理解业务逻辑和代码逻辑中的并发风险。

笔者的前一家公司因为初期业务量不大，代码中广泛使用了表级锁。但是随着业务规模的扩张，性能急剧下降，所带来的改造成本也急剧上升。在代码早期合理地考虑并发风险，不要激进地使用锁能为以后的业务增长留下充足的余量。

事务隔离是为了避免数据不一致的情况，比如上面提到的脏读、幻读和不可重复读。 不同隔离级别能够避免不同程度的数据不一致行为，参考下面的表。

| 隔离级别         | 脏读 | 不可重复读 | 幻读 |
| ---------------- | ---- | ---------- | ---- |
| READ UNCOMMITTED | Y    | Y          | Y    |
| READ COMMITTED   | N    | Y          | Y    |
| REAPATABLE READ  | N    | N          | Y    |
| SERIABLIZABLE    | N    | N          | N    |

（N: 不会发生  Y:会发生 ）



> Q:  既然有了隔离级别，为什么还需要行锁，表锁？
>
> A：隔离级别的差异只是为了解决特定的一种或几种数据不一致的可能， 除非你的数据库是串行执行事务的，否则幻读是你不可能避免的事情。隔离级别开的越高，数据库的并发性能越差。一般数据库的默认配置使用`READ COMMITTED`级别，避免脏读。那么为了避免不可重复读和幻读，你就需要在执行事务时，切换到更高隔离级别。 
>
> Q：不同数据库在隔离级别上有何差异？
>
> A： MySQL的默认隔离级别是`Repeatable Read`，PostgreSQL 的默认隔离级别是`Read Committed`。但是也取决于你的实际配置。需要特殊说的是：四种隔离级别虽然应该遵循ISO/IEC标准给出的四种隔离级别的定义。但是在不同数据库上，因其不同的实现方式而略有差异。当postgresql使用`Repeatable read`时，幻读事实上是不可能的，这不意味着：postgresql的RR隔离等同于seriablizable。postgresql的”可重复读“隔离仅查看到事务开始之前已提交的事务，不会看到正在并发执行的事务中未提交的数据或修改。但是，但是，查询确实能看到在事务内已经发生但尚未提交的操作。这种隔离比SQL标准所要的强得多。[参考](https://www.postgresql.org/docs/9.4/transaction-iso.html)
